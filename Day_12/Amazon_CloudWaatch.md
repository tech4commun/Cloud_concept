# Amazon CloudWatch — Getting Started Course Notes

> "Everything fails all the time." — Werner Vogels, Amazon CTO

## 1. Introduction to CloudWatch

**What it is**
CloudWatch is AWS's core **observability service** — it collects **metrics, logs, and events** from AWS resources and applications (in AWS, on-premises, hybrid, or other clouds) to give system-wide visibility into resource utilization, application performance, operational health, and user experience. It's part of AWS's broader observability offering, alongside APM and open-source solutions.

**Problems it solves**
Modern IT systems are complex and distributed — spanning multiple apps/services across AWS, on-prem, hybrid, and other clouds — making it hard to spot, diagnose, and resolve issues and understand their impact.

CloudWatch brings this data into **one place** to:
- Collect, search, aggregate, visualize, and respond to data about users, applications, infrastructure, and network
- Trigger notifications on errors via **alarms**
- Detect changes in response times
- Identify the impact and location of a problem
- Integrate with **AWS X-Ray** to connect traces with metrics and log data

**Key benefits**
- **Understand application health** — full-stack observability in one place; detect problems quickly, investigate efficiently, reduce **MTTR** (mean time to resolution)
- **Accelerate collaboration** — route alerts to the right stakeholders; automate repetitive tasks; take a user-centric approach
- **Reduce operational cost** — understand resource usage patterns; small efficiency gains scale across many instances; better capacity planning enables Reserved/Spot Instance savings
- **Increase customer satisfaction** — improve availability, reduce downtime, build faster/seamless digital experiences

**Pricing**
- Priced **per feature/capability and data type** — no single flat rate
- **No minimum fees or upfront commitments** — pay only for what you use
- **AWS Free Tier** eligible for many features
- Use the **AWS Pricing Calculator** to estimate costs

## 2. Architecture and Use Cases

### Approach 1: Outside-In (User Experience Focus)
Understand and improve the **end-user digital experience**, regardless of underlying infrastructure.

| Step | Tool | Purpose |
|---|---|---|
| 1 | **CloudWatch Synthetics** | Create **canaries** — scripted tests that mimic real user actions (e.g., load a page, search a product, add to cart, purchase). Scheduled to run continuously, even with zero real traffic, to catch page load errors and broken workflows early |
| 2 | **CloudWatch RUM** (Real User Monitoring) | Collects actual data from real users' browsers — latency, page errors, location, device type, browser — to spot client-side bottlenecks and regional/browser-specific issues |
| 3 | **CloudWatch Dashboard** | Combine Synthetics + RUM + other data into a custom, user-focused dashboard; view logs and metrics together with time controls for current/historic troubleshooting |
| 4 | **Notifications (Alarms + SNS)** | Set thresholds on key page latency; get notified immediately via **SNS** (e.g., email) when breached, so you can investigate before it escalates |

### Approach 2: Inside-Out (Infrastructure Scaling Focus)
Example: an e-commerce app on **EC2 instances behind an ELB**, with traffic that spikes (e.g., during a sale) — goal is to scale to meet demand without over-provisioning permanently.

| Step | Tool | Purpose |
|---|---|---|
| 1 | **CPUUtilization metric** | A default (vended) EC2 metric — view via CloudWatch console or the EC2 console's Monitoring tab, to understand server load |
| 2 | **CloudWatch Alarms** | Set a threshold on CPU utilization; trigger an action — in this case, **EC2 Auto Scaling** — when the threshold is breached |
| 3 | **CloudWatch Dashboard** | Ingest web server logs into CloudWatch Logs, extract page response time data, and visualize it alongside CPU utilization to confirm the scaling action actually improved user experience |

### Core Technical Concepts

| Concept | Description |
|---|---|
| **Metrics** | Time-ordered series of numerical data (e.g., EC2 CPU usage). Uniquely defined by **namespace** + **metric name** + **dimensions** (zero or more). AWS service metrics use namespaces like `AWS/EC2`. **Dimensions** are name/value pairs adding context (e.g., instance ID) and are used to filter/search. Many AWS services publish metrics automatically; you can also create **custom metrics**, including via the **CloudWatch agent** for EC2/on-prem/other-cloud servers |
| **Logs** | Text-based messages from apps/services — richer/more contextual than metrics, often structured (JSON, space-separated). Ingested via the CloudWatch agent (for servers) or native integrations (Lambda, CloudTrail, VPC Flow Logs). Use **metric filters** to extract metrics from log data |
| **Traces** | Data generated by a single request (e.g., an HTTP GET/POST) as it travels through a load balancer, app code, and downstream AWS/external API calls. The fundamental data type of **AWS X-Ray**, usable alongside CloudWatch logs and metrics |
| **Synthetics & Canaries** | Configurable scripts (canaries) that monitor URLs, REST APIs, and website content by following the same routes/actions as a customer — schedulable for continuous verification |
| **Metric Insights** | SQL-based query language for querying metrics **at scale** (has query limits vs. Metric Math) |
| **Metric Math** | Search + math functions (MAX, MIN, AVG, STDEV, IF, arithmetic/comparison/logical operators) for analyzing metric data over **longer time periods** (up to full metric history) |
| **Logs Insights** | Query language for searching/aggregating log data; results can be visualized on a dashboard as tables or various chart types |
| **Dashboards** | Visualize metrics and logs via customizable, resizable **widgets** — text, tables, charts, and alarm status — built per-persona or per-application |
| **Alarms** | Trigger actions when metric data crosses a threshold (static or **anomaly-detection-based**). States: `OK`, `ALARM`, `INSUFFICIENT DATA`. Actions include SNS notifications, EC2 actions, Systems Manager actions, Auto Scaling actions, and ticketing actions. A **metric alarm** = single metric or math expression resulting in one time series. A **composite alarm** = logically combines the state of multiple alarms via a rule expression |

> Both Metric Insights and Metric Math results can feed a dashboard or an alarm — but an alarm requires the query to resolve to a **single time series**.

### Typical Use Cases
1. **Deliver a unified observability experience** — end-to-end visibility across AWS, on-prem, hybrid, and other clouds; visualize multi-account data in one place
2. **Visualize data in context, at scale** — custom dashboards per app/role; surface SLOs, KPIs, user experience, performance, and resource utilization
3. **Support incident detection, resolution, and root cause analysis** — alert on thresholds/anomalies; correlate metrics/logs/traces; automate resolution to reduce MTTR
4. **Improve end-user experience** — combine APM + Synthetics + RUM + internet monitoring to catch bottlenecks before they hit customers, especially around traffic spikes, infra changes, and deployments
5. **Improve cost and resource optimization** — combine utilization data with performance/UX data; trigger automated actions to balance cost vs. performance

### Best Practices to Keep in Mind

| Practice | Guidance |
|---|---|
| **Identify success criteria** | Don't try to observe everything at once — pick a few key areas (e.g., MTTR, proactive detection, UX, resource usage) and define success criteria for those first |
| **Ingest data deliberately** | Don't ingest everything — it raises cost and makes finding relevant data harder. Filter irrelevant logs, set log retention policies, and think about metric frequency/cost. Decide whether data is better as logs or metrics |
| **Plan agent deployment** | Use **AWS Systems Manager** to manage CloudWatch agent rollout/config at scale — store configs in Parameter Store, use state associations, schedule updates, use tags/resource groups for new instances, or bake the agent into your AMI/user data |
| **Avoid alarm fatigue** | Only create alarms tied to a defined action/owner. Start with fewer, meaningful alarms. Link alarms into your existing incident management process |
| **Create context-rich dashboards** | Show trends over time, not just point-in-time alarms (a fast upward trend vs. a slow one needs different urgency). Add contextual info and links to docs/support teams |
| **Watch for PII** | Log data may contain sensitive info — mask/remove it before ingestion, or use **CloudWatch Log group data protection policies** to detect and mask sensitive data automatically |

### Quick Reference Summary
- **Metrics** = numeric time series; **Logs** = detailed text events; **Traces** = per-request journey (via X-Ray)
- **Synthetics** = scripted "fake user" tests (canaries); **RUM** = real user browser data
- **Metric Insights** (SQL, scale-limited) vs. **Metric Math** (functions, longer history)
- **Alarms**: metric alarm (single time series) vs. composite alarm (combines multiple alarm states)
- Outside-in = user experience monitoring (Synthetics → RUM → Dashboard → Alarms/SNS)
- Inside-out = infrastructure scaling (CPUUtilization → Alarm → Auto Scaling → Dashboard validation)
- Best practice: start small, avoid alarm fatigue, manage PII, deploy agents via Systems Manager
