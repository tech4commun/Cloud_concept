# Core AWS Storage Services — Notes

## AWS Storage Portfolio — Overview

AWS Storage = core services + closely associated services. Pick what fits the use case; combine services instead of hosting/maintaining separate systems yourself.

- Pay only for what you use → cheap to test apps, use cases, or alternate storage services, then delete when done.
- **All storage is private by default** — you explicitly grant access (public, other AWS services, other accounts).

### 3 Core Storage Types
| Type | Analogy | Best For |
|---|---|---|
| **Block** | DAS / SAN | Databases, ERP — dedicated, ultra-low-latency, per-host storage |
| **File** | NAS | Shared files — content repos, dev environments, media stores, home directories |
| **Object** | — | Scalable, metadata-rich storage — modern apps, analytics, backup, archive |

## Block Storage

### EC2 Instance Store
- Ephemeral, physically attached to host — direct-attached-disk architecture.
- Sub-millisecond latency; only certain instance types support it.
- **Use for:** buffers, caches, scratch data, fleet-replicated data.
- **Data lost on:** disk failure, stop, hibernate, terminate. **Survives:** reboot only.
- Not recommended for most workloads — no replication/durability guarantees.

### Amazon EBS — Overview
- High-performance block storage for EC2, throughput- and transaction-intensive workloads at any scale.
- Raw, unformatted block device — mount, format, use like a hard drive.
- Persists **independently** of instance life; **dynamically resizable** (unlike fixed physical disks).
- Replicated within an **AZ**; scales to **petabytes** across multiple volumes.
- Snapshots + lifecycle policies → back up to S3, geographic protection.
- Pay only for what you provision.

### EBS Features
| Feature | Key Detail |
|---|---|
| **Single-AZ** | Volume + instance must be in same AZ; use snapshots to move across AZs/Regions |
| **Persistent** | Survives instance termination (root volumes are the default exception, but can be made persistent); can detach/reattach to a different instance |
| **Volume types** | SSD (General Purpose gp2/gp3, Provisioned IOPS io1/io2) for transactional workloads; HDD (Throughput Optimized st1, Cold sc1) for throughput/low-cost workloads |
| **Elastic Volumes** | Resize, change type, tune performance — live, no downtime |
| **High availability/durability** | 99.8–99.9% durability (0.1–0.2% AFR) standard; **io2** offers 99.999% (0.001% AFR) — ideal for business-critical apps |
| **Encryption** | At-rest, in-transit (EC2↔EBS), and snapshots — all encrypted when enabled |
| **Native snapshots** | Point-in-time, stored in S3; restore to new volumes, expand, or move across AZs; automate via Amazon DLM |
| **AWS Backup support** | Centralized, policy-based backup automation |
| **Performance monitoring** | Bandwidth, throughput, latency, queue length via CloudWatch |

**Use cases:** enterprise apps, relational DBs, NoSQL DBs, big data analytics, file systems/media, **lift-and-shift** migrations (via AWS MGN / AWS SMS).

**Pricing:** based on volume type, size, provisioned IOPS/throughput; varies by AZ. Snapshots billed by actual storage used.

## File Storage — Overview

5 managed file services (or self-manage via EC2 + EBS):

| Service | Built On | Protocol(s) | Best For |
|---|---|---|---|
| **Amazon EFS** | AWS-native | NFS | Linux, cloud-native, multi-AZ elastic file storage |
| **FSx for Lustre** | Lustre | POSIX (Lustre client) | HPC, ML — extreme throughput/IOPS |
| **FSx for Windows File Server** | Windows Server | SMB | Windows apps, Microsoft workloads |
| **FSx for NetApp ONTAP** | NetApp ONTAP | iSCSI (block) + NFS/SMB (file) | Multi-protocol, existing NetApp shops |
| **FSx for OpenZFS** | OpenZFS | NFS (v3/v4.0/v4.1/v4.2) | ZFS/Linux migrations, high IOPS at low latency |

> Self-managed option: EC2 + EBS running your own file system (SMB, NFS, XFS, GPFS, ZFS, or custom).

**Payment models:** EFS & S3 = pay for capacity **consumed**. EBS, FSx for Lustre, FSx for Windows = pay for capacity **provisioned**.

## Amazon EFS

**Definition:** Simple, serverless, elastic file system — scales to petabytes automatically, no capacity planning.

- Protocol: **NFSv4.1 / NFSv4.0**.
- Accessible from: EC2, Lambda, ECS, EKS — shared data source across compute types.
- Pay only for storage used, no minimum fee.

### Storage Classes
| Class | Resilience |
|---|---|
| EFS Standard / Standard-IA | Multi-AZ |
| EFS One Zone / One Zone-IA | Single-AZ (cheaper) |

### Performance & Throughput Modes
- **Performance modes:** General Purpose (low latency, default) vs. Max I/O (higher aggregate throughput/IOPS, slightly higher per-op latency, Standard classes only).
- **Throughput modes:** Bursting (scales with size) vs. Provisioned (set independently of size).

### Key Features
- **Fully managed** — no server/hardware/patching/backup management.
- **11 nines durability**; Standard classes replicate across multiple AZs automatically.
- **Lifecycle management** — auto-moves files Standard → Standard-IA (or One Zone → One Zone-IA) after 7/14/30/60/90 days; up to **92% cost savings**.
- **Security:** VPC security groups (network), IAM + access points (app-level), POSIX permissions.
- **Scalable performance:** up to >10 GB/s throughput, >500,000 IOPS for top workloads.
- **Encryption:** at rest (KMS) and in transit (TLS).
- Integrates with containers/serverless (ECS, EKS, Fargate, Lambda) for stateful workloads.
- Data transfer/backup: **DataSync**, **AWS Backup**, **AWS Transfer Family** (SFTP/FTPS/FTP).

**Use cases:** container/serverless persistent storage, lift-and-shift enterprise apps, analytics/ML (SageMaker integration), web serving/CMS, dev/test environments, media workflows, database backups.

**Pricing:** pay for storage used (+ IA reads, + provisioned throughput); no minimum/setup fee.

## Amazon FSx for Lustre

**Definition:** Managed high-performance Lustre file system — for speed-critical workloads (ML, HPC, video processing, financial modeling).

- Delivers **hundreds of GiB/s** throughput, **millions of IOPS**.
- **POSIX-compliant** — Linux apps work unmodified; read-after-write consistency, file locking.
- **Native S3 integration** — presents S3 objects as files; write results back to S3.
- Scales to tens of thousands of CPU cores accessing the same storage simultaneously.

### Key Features
- **High/scalable performance:** parallel file system, linear scaling; SSD (latency/IOPS-optimized) or HDD (throughput-optimized, with optional SSD cache) storage.
- **Seamless data repository access:** native drive on EC2 Linux/EKS; imports directly from on-prem via Direct Connect/VPN (cloud bursting).
- **Simple/fully managed:** AWS provisions, configures, and maintains the Lustre software.
- **Cost optimized:** choose **scratch** (short-term processing) vs. **persistent** (longer-term) file systems; launch/delete in minutes.
- **Secure/compliant:** encrypted; PCI DSS, ISO, HIPAA eligible; VPC security groups + IAM + CloudTrail.

**Use cases (Horizontal):** machine learning (SageMaker integration), HPC (AWS Batch, AWS ParallelCluster).
**Use cases (Vertical):** genomics/life sciences, media processing/transcoding, autonomous vehicle simulation, SAS Grid computing.

**Pricing:** usage-based, no minimum fee. Scratch = GB-months of storage. Persistent = GB-months + SSD/HDD performance tier. Backups billed by actual data backed up.

## Amazon FSx for Windows File Server

**Definition:** Fully managed, native Windows file servers — SMB protocol, integrates with Active Directory.

- Highly durable/available, spans multiple AZs.
- Accessible from thousands of compute instances.
- Windows features + performance for easy lift-and-shift of enterprise Windows apps.

### Key Features
- **Built on Windows Server:** admin features (user file restore, quotas, ACLs); Active Directory integration (on-prem AD or AWS Managed Microsoft AD).
- **Broadly accessible:** EC2, ECS, VMware Cloud on AWS, WorkSpaces, AppStream 2.0; supports Windows Server 2008+/Windows 7+, Linux, macOS; on-prem via Direct Connect/VPN; cross-VPC/account/Region via Peering or Transit Gateway.
- **Fully managed:** provisioning, replication, failover/failback, Windows Server updates — all handled by AWS.
- **Performance/scale:** multi-GB/s throughput, hundreds of thousands of IOPS, sub-ms latency; up to **64 TB** per file system; DFS namespaces for scale-out beyond that.
- **Security:** encryption at rest (KMS) + in transit (SMB Kerberos); PCI DSS, ISO, SOC 1/2/3, HIPAA-eligible; ACLs, VPC security groups, IAM, CloudTrail; file access auditing via Windows Event Logs → CloudWatch/Kinesis.
- **Availability:** Single-AZ or **Multi-AZ** (active + standby, synchronous replication, automatic failover); supports HA SQL Server via Continuously Available (CA) file shares.
- **Cost:** HDD (general-purpose) or SSD (latency-sensitive) storage; **data deduplication** (50-80% typical savings depending on data type); user quotas.
- Migration via **AWS DataSync**.

**Use cases:** home directories, lift-and-shift Windows apps (ERP/CRM), HA SQL Server, media workflows, web serving/CMS (IIS/WordPress), data analytics.

**Pricing:** pay for storage + throughput provisioned + backups + cross-AZ/Region data transfer; no minimum fee.

## Amazon FSx for NetApp ONTAP

**Definition:** Fully managed NetApp ONTAP file system — familiar NetApp features/APIs with AWS-managed simplicity.

- **Multi-protocol:** iSCSI (block), NFS + SMB (file) — even **concurrent** NFS+SMB access to the same data.
- SSD storage, sub-millisecond latency.
- One-click snapshot, clone (**FlexClone**), and replicate (**SnapMirror**).
- Automatic tiering to lower-cost storage — pay SSD price for only the active fraction of data.
- Ideal for on-prem NetApp customers migrating, backing up, or bursting to AWS with **no app changes**.

### Key Features (7 categories)
1. **Administration/Management:** AWS-native or NetApp tools (Console, CLI, SDK + NetApp Cloud Manager, ONTAP REST API); **SnapMirror** for fast migration; **FlexClone** for instant, storage-efficient point-in-time clones.
2. **Accessibility:** multi-protocol (NFS/SMB/iSCSI); Multi-AZ option (active + standby, synchronous replication, auto-failover); access from EC2, ECS, EKS, VMware Cloud, WorkSpaces, AppStream.
3. **Performance/Scale:** multi-GB/s throughput, hundreds of thousands of IOPS; read replicas for read-heavy scaling; sub-ms (SSD) or tens-of-ms (capacity pool) latency; petabyte-scale per file system.
4. **Cost Optimization:** two-tier storage — **primary (SSD)** + **capacity pool (elastic, low-cost)**, auto-tiered by access pattern; compression + dedup (~65% typical savings); thin-provisioned volumes; independent throughput/IOPS selection.
5. **Availability/Data Protection:** NetApp Snapshots (file/folder restore); **SnapMirror cross-Region replication** (RPO as low as 5 min, RTO single-digit minutes); automated daily incremental, crash-consistent backups.
6. **Hybrid Workflow Support:** on-prem caching (Global File Cache, FlexCache); backup/DR to AWS; cloud bursting via FlexCache as an in-cloud cache for on-prem data.
7. **Security/Compliance:** VPC security groups/NACLs + ONTAP export policies; IAM resource-level permissions; AD-based authentication; KMS encryption at rest + Kerberos in transit; CloudTrail logging; PCI DSS/ISO/SOC/HIPAA-eligible; file access auditing (FPolicy); antivirus (vscan) via AWS Partner tools.

**Use cases:** similar to EFS/FSx Windows/FSx OpenZFS, but distinguished by **multi-protocol** (NFS+SMB+iSCSI) access and NetApp migration scenarios.

**Pricing components:** SSD storage, SSD IOPS, capacity pool usage, throughput capacity, backups, cross-AZ/peering data transfer, capacity pool read/write requests.

## Amazon FSx for OpenZFS

**Definition:** Fully managed OpenZFS file system — replaces on-prem Linux/ZFS file servers with AWS-managed agility.

- Protocol: **NFS v3, v4.0, v4.1, v4.2**. (SMB not supported at initial launch.)
- Powered by **AWS Graviton** processors + latest disk/network tech.
- Up to **1 million IOPS**, latency in the hundreds-of-microseconds range.

### Key Features (6 categories)
1. **Performance/Scale:** up to 12.5 GB/s throughput & 1M IOPS (cached); up to 4 GB/s & 160,000 IOPS (persistent disk); thousands of simultaneous clients, multiple parallel connections per client.
2. **Cost Optimization:** independent storage/performance scaling; **Z-Standard compression**; multiple volumes per file system, thin provisioning, user/group quotas.
3. **Accessibility:** full NFS v3–v4.2 support; EC2, ECS, EKS, VMware Cloud, WorkSpaces, AppStream; cross-VPC/Region via Transit Gateway/Peering; on-prem via Direct Connect/VPN.
4. **Administration/Management:** native ZFS snapshots + **instant data cloning** (via FSx API); manage via Console/CLI/SDK; integrates with IAM, CloudWatch, CloudTrail, CloudFormation.
5. **Availability/Data Protection:** automatic daily backups to S3 (incremental, cross-Region copyable); near-instant point-in-time snapshots stored locally for fast file-level restore.
6. **Security/Compliance:** KMS encryption at rest, 256-bit in-transit encryption (supported EC2 types); IAM + resource tagging; CloudTrail + GuardDuty API monitoring; POSIX permissions/ACLs; PCI DSS/ISO/SOC/HIPAA-eligible.

**Use cases:** seamless workload migration, faster data analytics insights, accelerated content management, increased dev/test velocity.

**Pricing components:** SSD storage, SSD IOPS, throughput capacity, backup storage; billed per-second, prorated monthly. Plus cross-AZ/Region data transfer.

## Amazon S3 (Object Storage)

**Definition:** Object storage for the internet — industry-leading scalability, availability, security, performance. **11 nines (99.999999999%) durability.**

- Data stored as **objects** in **buckets**; flat, non-hierarchical structure (uses prefixes for organization).
- Max object size: **5 TB**.
- Access via S3 Access Points or bucket hostname; standards-based **REST API**.
- **Performance:** parallel requests scale with prefixes — at least **3,500 req/s** (PUT/write) and **5,500 req/s** (GET/read) *per prefix*; virtually unlimited prefixes.
- **Consistency:** strong read-after-write consistency for PUT/DELETE (new and overwritten objects) in all Regions; also strongly consistent for S3 Select, ACLs, tags, metadata (HEAD).

**Use cases:** data lakes, websites, mobile apps, backup/restore, archive, enterprise apps, IoT, big data analytics.

### Storage Classes
| Class | Use Case |
|---|---|
| **S3 Standard** | General-purpose, frequently accessed |
| **S3 Standard-IA** | Infrequent access |
| **S3 One Zone-IA** | Infrequent access, lower availability need, cheaper |
| **S3 Intelligent-Tiering** | Unknown/changing access patterns — auto-moves between tiers |
| **S3 Glacier Instant Retrieval** | Lower-cost archive, instant retrieval |
| **S3 Glacier Flexible Retrieval** | Low-cost archive, minutes–hours retrieval |
| **S3 Glacier Deep Archive** | Lowest cost, retrieval up to 12 hours |
| **S3 on Outposts** | On-prem hybrid, data residency requirements |

> **S3 Lifecycle policies** automate transitions between classes (and expirations) — no app changes needed. **S3 Storage Class Analysis** identifies what should move to cheaper tiers.

### Management Features
- **Storage management:** bucket names, prefixes, tags, S3 Inventory; **S3 Batch Operations** (copy, tag, ACL changes, Glacier restores, Lambda-driven custom processing) at massive scale.
- **Version control:** S3 Versioning (preserve/retrieve/restore every version); **MFA Delete** to prevent accidental deletion.
- **Replication:** Cross-Region Replication (CRR) and Same-Region Replication (SRR); **S3 RTC** (Replication Time Control) gives SLA-backed replication timing.
- **Retention/Compliance:** **S3 Object Lock** (WORM) — Governance mode (IAM-removable) vs. Compliance mode (immutable, no removal by anyone); tracked via S3 Inventory.
- **Monitoring:** cost allocation tags + Cost Allocation Reports; CloudWatch (health + billing alarms); CloudTrail (bucket/object activity); **S3 Event Notifications** → trigger Lambda/workflows.

### Analytics & Insights
- **S3 Storage Lens** — org-wide usage/activity visibility + automated optimization recommendations; drill-down to org/account/Region/bucket/prefix level.
- **S3 Storage Class Analysis** — access-pattern analysis to inform Lifecycle policy tuning.

### Access Management & Security
- **Access control:** IAM, ACLs, bucket policies, **S3 Access Points** (per-app access), Query String Authentication (temporary URLs); audit logs for all requests.
- **On-prem connectivity:** VPC endpoints, **AWS PrivateLink for S3** (private access over Direct Connect/VPN).
- **Encryption:** default bucket encryption — SSE-S3 (S3-managed keys) or SSE-KMS (AWS KMS keys); **S3 Bucket Keys** reduce SSE-KMS request costs.
- **S3 Block Public Access:** account- or bucket-level controls that override other permissions to enforce "no public access"; backed by Trusted Advisor, CloudTrail, CloudWatch.
- **Access Analyzer for S3:** flags buckets with unintended access; exportable CSV findings; IAM "last accessed" data to trim unused permissions.
- **Amazon Macie:** ML-based discovery of sensitive data (PII) in S3; flags public/unencrypted/externally-shared buckets; integrates with EventBridge for automated remediation.

### Data Processing & Query
- **S3 Object Lambda:** runs a Lambda function on GET request output — on-the-fly data transformation.
- **Query in place:**
  - **S3 Select:** SQL-based partial object retrieval — up to 400% faster queries, up to 80% cost reduction (vs. fetching whole 5 TB objects).
  - **Amazon Athena:** serverless SQL directly on S3 data, fast ad hoc discovery.
  - **Amazon Redshift Spectrum:** SQL on S3 data at rest, suited for complex queries at exabyte scale; shares data catalog/format with Athena.

**Use cases:** backup/restore, disaster recovery, archive (Glacier/Deep Archive — retrieval as fast as 1 min expedited, 3-5 hrs standard, 12 hrs for Deep Archive), data lakes/big data (+ AWS Lake Formation), hybrid cloud storage (PrivateLink, Storage Gateway, DataSync, Transfer Family), cloud-native application data.

**Pricing components:** storage, request/data retrieval, data transfer/transfer acceleration, data management/analytics, S3 Object Lambda processing. Varies by Region.

## Quick Comparison: File Storage Services

| Service | Protocol | AZ Model | Best Fit |
|---|---|---|---|
| **EFS** | NFS | Multi-AZ (Standard) / Single-AZ (One Zone) | Linux, cloud-native, elastic scale |
| **FSx for Lustre** | POSIX (Lustre) | Single or multi | HPC/ML — extreme throughput |
| **FSx for Windows File Server** | SMB | Single-AZ or Multi-AZ | Windows/Microsoft workloads |
| **FSx for NetApp ONTAP** | iSCSI + NFS + SMB | Multi-AZ option | Multi-protocol, NetApp migrations |
| **FSx for OpenZFS** | NFS only (at launch) | — | ZFS/Linux migrations, low-latency IOPS |

## Quick Reference: Payment Models
- **Pay for consumption:** EFS, S3
- **Pay for provisioned capacity:** EBS, FSx for Lustre, FSx for Windows File Server (and generally FSx for ONTAP/OpenZFS components too)
